# Risk Register

Status: technical/product readiness draft, 2026-06-28. Not legal advice. Must be reviewed by a German privacy lawyer or external DSB before public launch.

| Risk ID | Area | Description | Severity | Likelihood | Current status | Mitigation | Owner/TODO | Must fix before public launch |
|---|---|---|---|---|---|---|---|---|
| R-001 | Uploads/storage | The target Supabase project could diverge from the locally tested private bucket policies. | High | Medium | Versioned migration and local two-user Storage tests pass; target state is unverified. | Apply migrations, verify buckets/advisors and repeat the QA on the intended beta project. | Engineering + DSB review | Yes |
| R-002 | Local uploads | Static `/uploads` file serving can expose uploaded documents if enabled in production. | High | Medium | Static mount is disabled by default and only enabled by `AVARENO_ENABLE_STATIC_UPLOADS`; UI uses short-lived signed API download tickets. | Keep static uploads disabled in production and move real private files to private object storage. | Engineering | Yes |
| R-003 | Third-country transfer | US/third-country provider transfer status for Supabase, Cloudflare, AI, email, analytics, support is unknown. | High | Medium | Provider facts not verified. | Complete DPA/AVV and transfer mechanism review. | Legal/DSB | Yes |
| R-004 | DPA/AVV | DPA/AVV status is unknown for core providers. | High | High | Unknown. | Verify and store signed agreements outside repo. | Founder/legal | Yes |
| R-005 | Export | MVP JSON/ZIP export bundle exists for local data, but production export jobs, provider-side exports and cross-user production tests are incomplete. | High | High | Partial MVP controls. | Build production export job flow including private storage files, signed/authenticated downloads, provider-side receipts and cross-user tests. | Engineering | Yes |
| R-006 | Deletion | Deleted data may survive in future providers or backups even though the active account deletion path succeeds. | High | Medium | Local/Supabase database, Storage, files and Auth deletion are integrated and tested; backups/future connectors remain open. | Repeat target-project QA; define backup expiry/restore exclusions and provider revocation before enabling connectors. | Engineering | Yes |
| R-007 | Logs | API/server/host logs may contain sensitive query strings, filenames, document text, tokens, or connector payloads. | High | Medium | Safe audit helpers exist, but host/API log policy is incomplete. | Add no-sensitive-logs checks, scrubbers, retention policy, monitoring config. | Engineering | Yes |
| R-008 | AI prompts | Future AI prompts may contain personal documents/receipts or sensitive Vault content. | High | Medium | Current AI/OCR is mock; guardrails exist. | Explicit opt-in, payload minimization, Vault block, provider DPA/AVV/no-training review. | Product + Engineering + Legal | Yes |
| R-009 | Vault AI | Automatic analysis of sensitive Vault documents could process identity/insurance/medical/legal docs without explicit consent/control. | High | Medium | Guardrail constant blocks category auto-analysis in extraction path; full Vault not implemented. | Keep Vault no-auto-analysis default, add re-auth, explicit action and audit trail. | Engineering/Product | Yes |
| R-010 | Connector tokens | Connector tokens/secrets could be stored insecurely if Connect evolves. | High | Medium | Secret storage intentionally raises/refuses plaintext in helper. | Implement encryption, rotation, revocation, disconnect and no frontend exposure. | Engineering | Yes |
| R-011 | SSRF | Custom connector/provider URLs could access localhost, metadata or internal networks. | High | Medium | SSRF validation helpers exist and are used in compatibility sandbox fetch. | Add tests, allowlists, response limits, rate limits, apply helper to every outbound connector. | Engineering | Yes |
| R-012 | Bulk imports | Importing entire mailbox/drive/archive could violate minimization and create high-risk sensitive data processing. | High | Medium | Not implemented; AGENTS prohibits. | Selected imports only, preview, scope UI, no default bulk import. | Product/Engineering | Yes |
| R-013 | Cookies/analytics | Nonessential cookies/analytics could be added without valid consent/disclosure. | High | Medium | No analytics found; Turnstile optional. | Consent review before tracking, maintain cookie inventory. | Product/Legal/Engineering | Yes |
| R-014 | Affiliate marking | Affiliate links/clicks may not be clearly marked to users. | Medium | Medium | Affiliate click table and UI fields exist. | Mark affiliate/reorder links, define retention and disclosure. | Product/Legal | Yes |
| R-015 | Newsletter | Newsletter could launch without double opt-in/unsubscribe/retention record. | Medium | Low | No implementation found. | Select provider, double opt-in, unsubscribe, consent history. | Product/Legal | Yes if newsletter enabled |
| R-016 | Discord/support | Users may post sensitive documents/screenshots in public support/community channels. | High | Medium | No official support tooling found; AGENTS warns. | Use private support flows, public warnings, screenshot redaction guidance. | Product/Support | Yes |
| R-017 | Public legal pages | Impressum/Datenschutz pages need final legal review before launch. | High | Medium | Supplied SelaPrinting Studio / Stefan Weiss operator details are added; full legal review remains open. | Verify contact requirements, final Datenschutz wording, provider facts, AGB/Widerruf and legal review. | Founder/legal | Yes |
| R-018 | Storage public-by-accident | Avatars are public by design; target-project drift could expose private document buckets. | High | Medium | Versioned private-bucket migration and local tests pass. | Verify target dashboard and run automated User A/User B/anon tests before invitations. | Engineering | Yes |
| R-019 | RLS misconfiguration | A target-project migration or grant mismatch could bypass tested RLS. | High | Medium | Versioned forced-RLS migration and 56-check local regression pass. | Apply migrations, run advisors and repeat cross-user QA on the target project. | Engineering | Yes |
| R-020 | Backups | Deleted data may remain in backups too long or be restored without deletion reconciliation. | High | Medium | Backup policy unknown. | Define backup retention, restore procedure, deletion tombstone/replay. | Engineering + Legal | Yes |
| R-021 | Search | Search reads extracted text and reminder content; sensitive queries may enter logs. | Medium | Medium | Query not persisted in app code. | Avoid logging query strings, scrub server logs, use POST for sensitive search if needed. | Engineering | Before public beta |
| R-022 | Upload validation | Upload endpoint still lacks malware scanning and deep content validation. | High | Medium | MVP max-size, extension allowlist and MIME allowlist are implemented. | Add malware scanner strategy, production private storage and deeper content validation where needed. | Engineering | Yes |
| R-023 | Supabase auth metadata | Authorization based on user-editable metadata would be unsafe if added later. | High | Medium | Current backend uses `sub`; profile uses metadata for UI only. | Keep authz in DB/app metadata, not `user_metadata`; document RLS rules. | Engineering | Yes |
| R-024 | SmartThings token | Local env token path could become production token storage pattern. | High | Medium | Env token for local testing. | Replace with OAuth and encrypted server-side token store before production. | Engineering | Yes |
| R-025 | AI/legal claims | Warranty/insurance/support AI output could be perceived as guaranteed. | Medium | Medium | Guardrails mention not guaranteed. | Keep user-confirmed source/provenance and no legal/insurance guarantee claims. | Product/Legal | Yes |
| R-026 | Billing/provider review | Stripe is planned for subscription billing, but Tax/VAT handling, invoice duties, DPA/AVV, region/transfers, retention and cancellation flows are not yet verified. | High | Medium | Server-side Checkout, portal and webhook foundation added. | Complete legal/tax/provider review, update privacy policy/terms, and test Checkout, portal and webhook processing before paid launch. | Founder + Legal/Tax + Engineering | Yes |
| R-027 | Payment webhook integrity | Billing webhooks could incorrectly change subscription state if accepted without signature/idempotency. | High | Low | `/api/webhooks/stripe` requires `Stripe-Signature` and stores provider event ids for idempotency. | Verify against Stripe test events without logging raw payloads. | Engineering | Yes |
